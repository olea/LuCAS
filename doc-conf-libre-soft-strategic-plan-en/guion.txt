<h2>Talk targets<p>
<ul>

	<li>To show a simple strategic model to ensure the libre software success.
	<li>To show the solution guidelines.

<h2>Some of the libre software successes<p>
<ul>
	<li>An almost complete operating system available in several flavors.
	<li>At least two almost complete GUI.
	<li>At least one complete office suite.
	<li>Other key software...
	<li>A great developers community: programmers and computer and net resources.
	<li>The bigger interest at the moment: Industry, Market and Public Administrations <font size=+1>want</font> Libre Software.
</ul>

<h2>The main libre software weakness<p>
<ul>
	<li>Lack of libre equivalents of widely used applications.
	<li>Lack of QA.
	<li>Lack of standardization: it's not trivial to interchange a free app for another free equivalent.
	<li>Libre software projects use to be managed only by programmers:
	<ul>
		<li>Programmers loves to code the system but not the high level applications.
		<li>Programmers loves to write their own implementation from scratch.
		<li>Programmers easily forget the service ethic: they work for themselves and not for the final user.
	</ul>
</ul>

	<h2>Conclusions:</p> 
	<ol>
	<li>If libre software can't reach the requisites and needs of the present interested users (Industry, Market and Public Administrations) possibly we'll lost our best and definitely opportunity.
	<li>We need a strategical point of view which complete and strengthen the programmer's work.
	</ol>

<h2>Introducing into the Open Systems Software Architecture<p>
<center><img src="open-systems-architecture.png" alt="open systems architecture diagram"></center>
<ul>
	<li>All made with libre software.
	<li>Nothing to do with the 80's-90's open system concept
	<li>Formulated by Francisco de Urquijo (from Mexico)
</ul>

<h2>Detail of the Open Systems Software Architecture<p>
<dl>
	<dt><b>Standard Base Operating System</b>: the data infrastructure
	<dd><ul>
			<li>The operating system
			<li> GUI environment
			<li>office suites
			<li> etc.
		</ul>
	<dt><b>Standard Organizational Operating System</b>: the information infrastructure
	<dd>The integration of:<ul>
			<li>Work-group/collaboration environment.
			<li>ERP.
			<li>Work-flow environment.
		</ul>
	<dt><b>Standard Knowledge Operating System</b>: the knowledge infrastructure
	<dd><ul>
			<li>Modular, applicable and reusable knowledge.
			<li>Without doubt, the next step in Computer Sciences and user application.
		</ul>
</dl>



<h2>Status of the Open Systems Software Architecture implementations<p>
<dl>
	<dt><b>Standard Base Operating System</b>
	<dd><ul><li>Almost complete</ul>
	<dt><b>Standard Organizational Operating System</b>
	<dd>Partial implementations but no integration
	<ul>
			<li>Work-group/collaboration environment: lot's of tools available.
			<li>ERP: almost anything usable.
			<li>Work-flow environment: almost anything.
		</ul>
	<dt><b>Standard Knowledge Operating System</b>
	<dd><ul>
			<li>Almost inexistent in the actual market: neither
		</ul>
</dl>



<h2>To be done in the Open Systems Software Architecture implementations (1): Research & Development<p>
<dl>
	<dt><b>Standard Base Operating System</b>
	<dd>
	 It needs an Q&A effort.
	<dt><b>Standard Organizational Operating System</b>
	<dd>
		It needs Integration and Development efforts.
	<dt><b>Standard Knowledge Operating System</b>
	<dd>
		It needs a Research & Development effort.
</dl>



<h2>To be done in the Open Systems Software Architecture implementations (2): Methodology<p>
<ul>
	<li>Definition and application of Quality Assurance.
	<li>Definition and application of Standars on modular/interchangable  applications.
	<li>Spreading results in the Community, Industry, Market and Public Administrations.
</ul>

<h2>To be done in the Open Systems Software Architecture implementations (3): Key technologies<p>
<ul>
	<li>New development enviroment: Mono, XML-RPC/SOAP, etc.
	<li>UML analysis/design tools and UML compilers
	<li>Self managed applications: p2p.
</ul>
