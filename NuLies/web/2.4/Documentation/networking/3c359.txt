Nota:
Este controlador NO funcionará con las tarjetas 3C339 Token Ring; necesitará utilizar en su lugar el tms380.

Opciones:

El controlador acepta tres opciones: ringspeed, pkt_buf_sz y 
message_level.

Estas opciones pueden especificarse diferentemente para cada tarjeta 
encontrada.

ringspeed: Esta característica (la velocidad del anillo) está
ajustada a uno de estos valores: 0 (predeterminado), 4 ó 16. 0 hará que
la tarjeta detecte automáticamente la velocidad del anillo y se
configure a dicha velocidad; esta opción será la opción por defecto
para la mayoría de la gente. 4 ó 16 le permiten forzar la tarjeta para
funcionar a cierta velocidad. La tarjeta fallará si intenta
insertarla a una velocidad equivocada (a pesar de que algunos hubs
le permitirán hacer esto, así que tenga mucho cuidado). El propósito principal
de poner explícitamente la velocidad del anillo es para cuando la tarjeta es la
primera del anillo. En modo automático, si la tarjeta no puede
detectar ninguna actividad en el anillo se abrirá a la misma velocidad
a la que se abrió la última vez. Esto puede ser algo peligroso si
esta velocidad no coincide con aquella a la que usted quiere que
funcione el anillo.

pkt_buf_sz: Esta característica es el tamaño inicial reservado para el
búfer de recepción. Por defecto será de 4096 si no se indica
ningún valor. Puede incrementar el rendimiento del controlador
poniendo aquí un valor más grande que el tamaño de paquetes de la
red, a pesar de que el controlador ahora ajusta el tamaño de los
búferes basados en los valores de MTU.

message_level: Controla el nivel de mensajes creados por el
controlador. El predeterminado es 0: sólo visualiza los mensajes de arranque
y los críticos. Actualmente cualquier valor que no sea cero hará que se
visualicen todos los mensajes. Nota: Esto no activará los mensajes de
depuración. Para ello ha de modificarse el código fuente.

Tamaño de MTU variable:

El controlador puede manejar un tamaño de MTU de hasta 4.500 o 18.000
dependiendo de la velocidad del anillo. El controlador también cambia
el tamaño de los búferes recibidos como parte del reajuste del tamaño del mtu,
de modo que si pone mtu = 18.000, necesitará poder reservar
16*(sk_buff con tamaño del búfer de 18.000) que serán 18.500 bytes por
posición en el anillo = 296.000 bytes de espacio de memoria, más la
que se necesite para el sk_buff de tx. Recuerde que esto se hace por cada
tarjeta, es decir, que si está utilizando routers, gateways, etc., podría
empezar a usar un montón de memoria muy rápidamente.
